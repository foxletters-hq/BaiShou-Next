# 白守 Next — 打包与发布

本文档说明 **Android / Windows** 官方正式包的**本地构建与发布**流程。密钥与 `key.properties` **不入库**。

## 平台支持说明

受限于团队精力与开发、测试环境，**官方仅维护并发布 Android 与 Windows 客户端**：

| 平台            | 官方支持                                                                      |
| --------------- | ----------------------------------------------------------------------------- |
| **Android**     | 提供正式 APK，持续维护                                                        |
| **Windows**     | 提供 Inno Setup 安装包，持续维护                                              |
| **Linux**       | 不提供官方安装包与适配；有能力的用户可自行编译（见 `apps/desktop/README.md`） |
| **iOS / macOS** | 暂无计划，不提供安装包与适配                                                  |

下文发版流程仅涵盖 **Android + Windows**。`pnpm release:desktop:linux` 等命令仅供自行编译参考，**不会**纳入官方 Release 附件。

## 1. Android 签名

与旧版 Flutter 客户端（[BaiShou](https://github.com/Anson-Trio/BaiShou)）共用同一套 upload 证书，包名 `com.baishou.baishou`，用户可覆盖升级。

### 1.1 本地首次配置

```bash
# 从旧仓库复制（若你本机有 ../BaiShou/android/key.properties）
pnpm release:setup-signing
```

或参考 `apps/mobile/key.properties.example` 手动创建：

`apps/mobile/android/key.properties`（已 gitignore）

```properties
storePassword=...
keyPassword=...
keyAlias=upload
storeBase64=...   # 本地推荐：JKS 的 Base64，无需保留物理 .jks
```

### 1.2 本地打 Release APK

```bash
pnpm release:android
```

产物：`release/BaiShou-v{版本}-Android.apk`

### 1.3 CI Secrets（勿写入仓库）

在 **BaiShou-Next** 仓库 `Settings → Secrets and variables → Actions` 配置（与旧版 [BaiShou](https://github.com/Anson-Trio/BaiShou) 仓库 **同名、同值** 复制一份）：

| Secret                    | 说明                          |
| ------------------------- | ----------------------------- |
| `ANDROID_KEYSTORE_BASE64` | upload-keystore.jks 的 Base64 |
| `ANDROID_KEY_ALIAS`       | 通常为 `upload`               |
| `ANDROID_KEY_PASSWORD`    | Key 密码                      |
| `ANDROID_STORE_PASSWORD`  | Keystore 密码                 |

> **注意**：CI 会在 `expo prebuild` 前写入 `key.properties`，但 prebuild 会清空不完整的 `android/` 目录。构建脚本会在 prebuild 后自动恢复签名配置；若恢复失败，会回退读取上述环境变量。发版产物会校验 **不得使用 Android Debug 证书**。

## 1.4 开发包 vs 正式包

|              | 开发（`pnpm dev:mobile:clear`） | 发布（`pnpm release:android`） |
| ------------ | ------------------------------- | ------------------------------ |
| 包名         | `com.baishou.baishou.dev`       | `com.baishou.baishou`          |
| 桌面名称     | 白守 Dev                        | 白守                           |
| 签名         | debug                           | 正式 upload                    |
| ABI          | 默认四架构（含模拟器 x86）      | **仅 `arm64-v8a`（真机）**     |
| 与正式版关系 | **可并存**，互不影响            | 可覆盖旧 Flutter 版            |

改完原生配置后需重编一次：`pnpm dev:mobile:clear`

### 1.5 Android 覆盖安装与「无法降级」

系统安装界面显示的 **版本名**（如 `1.0.14 → 1.1.0`）是 `versionName`；能否覆盖安装只看 **`versionCode`（整数）**，新包的 `versionCode` 必须 **严格大于** 已安装包。

- 清单：`apps/mobile/src/version.json` 同时维护 `version` 与 `versionCode`；发版前执行 `pnpm sync:app-version`（`release:android` 会自动跑）。
- 每次发版请将 `versionCode` **+1**（或跳到更大整数），不要只改版本名。
- 若仍提示无法降级：用 `aapt dump badging 你的.apk | grep version` 对比手机里已装包的 `versionCode`（`adb shell dumpsys package com.baishou.baishou | grep versionCode`）。常见原因是装到了**旧 APK**（`versionCode` 未随版本名一起升）或曾装过更高 `versionCode` 的测试包。

安装器上的 **图标来自待安装 APK 内资源**，不是桌面图标缓存。Release 构建前会同步 `app-brand-icon-source.png`；换图后请 `pnpm sync:icons` 再全量 `pnpm release:android`。

## 2. 桌面端（Electron · Windows）

Windows 使用 **Inno Setup**（与旧版 Flutter 客户端同款安装器，四语言向导）。

```bash
# Windows 安装包（Inno Setup → Program Files\BaiShou）
pnpm release:desktop:win
```

**Windows 前置**：本机需安装 [Inno Setup 6](https://jrsoftware.org/isinfo.php)，或 `choco install innosetup -y`（仅含英文；中/日语言文件已随仓库 `apps/desktop/inno-languages/` 提供）。  
构建前请**退出所有白守窗口**（含已安装到 `Program Files` 或自定义目录的副本；否则会 `Access denied`；`build:unpack` 会自动尝试结束 `BaiShou.exe`）。  
安装/卸载时若白守仍在运行，安装向导会提示：**由安装程序结束进程**、**手动退出后再检测**，或**取消**（不再静默强杀导致卡住，也不再无提示直接文件占用失败）。

构建流程：`electron-vite build` → `electron-builder --dir` → **`verify-desktop-pack`** →（可选 Authenticode 签名）→ `setup.iss`（ISCC）→（可选再签 Setup.exe）。

**注意**：`win-unpacked` 能打开 ≠ 安装包一定正确。Inno 只是把当时的 `win-unpacked` 复制进安装包；若只重编了解包目录、没重跑 `build:inno`，或安装的是旧版 Setup.exe，仍会启动崩溃。请始终执行完整 `pnpm release:desktop:win` 后再安装。

产物目录：`apps/desktop/dist/`  
文件名：`BaiShou-v{版本}-Windows-Setup.exe`

### 2.1 Windows 代码签名（Authenticode）

未签名的 Setup.exe / `BaiShou.exe` 会触发 Windows SmartScreen「未知发布者」警告；若用户曾安装过**已签名**版本、新包又未签名，部分环境下覆盖安装/信任链也会异常。官方发版应使用 **OV/EV 代码签名证书**。

本地或 CI 配置任一即可启用自动签名（`scripts/sign-windows.mjs`）：

| 变量                         | 说明                                                   |
| ---------------------------- | ------------------------------------------------------ |
| `WINDOWS_CERT_FILE`          | 本机 `.pfx` 路径                                       |
| `WINDOWS_CERT_PFX_BASE64`    | CI 用：PFX 的 Base64（GitHub Secret）                  |
| `WINDOWS_CERT_PASSWORD`      | PFX 密码                                               |
| `WINDOWS_CERT_TIMESTAMP_URL` | 可选，默认 DigiCert 时间戳                             |
| `WINDOWS_SIGN_REQUIRED=1`    | 未配置证书或签名失败时**构建失败**（正式发版建议打开） |

本机还需 [Windows SDK](https://developer.microsoft.com/windows/downloads/windows-sdk/) 中的 `signtool.exe`（或设置 `SIGNTOOL` 指向它）。

GitHub Actions `build-windows` 已读取上述 secrets；仓库变量 `WINDOWS_SIGN_REQUIRED` 可设为 `1` 强制签名。未配置时构建仍成功，但日志会提示跳过签名。

校验：

```powershell
Get-AuthenticodeSignature apps/desktop/dist/BaiShou-v*-Windows-Setup.exe
```

`Status` 应为 `Valid`。

### 2.2 Linux 自行编译（非官方发版）

官方不提供 Linux 安装包。若需在 Linux 上自行编译，可参考 `apps/desktop/README.md` 执行：

```bash
pnpm release:desktop:linux   # 输出 AppImage 至 apps/desktop/dist/，仅供自用
```

自行编译产物**不保证**功能完整性与后续兼容性，相关问题不在官方支持范围内。

## 3. 一键官方打包（Android + Windows）

```bash
pnpm release:all
```

| 平台    | 产物目录             | 文件名示例                         |
| ------- | -------------------- | ---------------------------------- |
| Android | `release/`           | `BaiShou-v1.1.0-Android.apk`       |
| Windows | `apps/desktop/dist/` | `BaiShou-v1.1.0-Windows-Setup.exe` |

> Windows 包需在 **Windows 本机**构建（Inno Setup 6）。在 Linux / WSL / macOS 下 `pnpm release:all` 仅打 Android，Windows 请在本机执行 `pnpm release:desktop:win` 或使用 GitHub Actions。

## 4. 发布到 GitHub Release（CI 分端发版）

日常流程：**功能 PR 合并到 main** → 仅 bump 需要发版的一端 `version.json` → 在 main 上打 **分端 tag** → CI 自动构建并发布。

### 4.1 版本号

| 端      | 清单                                               |
| ------- | -------------------------------------------------- |
| Android | `apps/mobile/src/version.json`（含 `versionCode`） |
| Windows | `apps/desktop/src/version.json`                    |

两端**可以不同版本**（例如 mobile 1.2.4、desktop 1.2.0）。仅改一端时，只 bump 该端清单。

```bash
pnpm sync:app-version
pnpm ci:check
```

### 4.2 Tag 约定（触发 CI）

| Tag              | 构建       | GitHub Release                                                                      |
| ---------------- | ---------- | ----------------------------------------------------------------------------------- |
| `mobile/v1.2.12` | 仅 Android | 挂到 `v{max(mobile,desktop)}`（例：桌面仍为 1.2.7 时 → `v1.2.12`）                  |
| `desktop/v1.2.7` | 仅 Windows | 同样挂到两端较大版本（例：移动已是 1.2.12 → 仍发布到 `v1.2.12`，追加 Windows 产物） |

分端 tag 仅用于**触发 CI**；GitHub Release 的展示/挂载版本取 **两端当前 version 的较大者**，后发平台追加到同一条 Release，避免出现「桌面 1.2.7、移动 1.2.12」两条分叉的 Latest。

```bash
# 合并到 main 后
pnpm release:tag mobile          # 预览命令
pnpm release:tag mobile --push   # 打 tag 并推送，触发 CI

pnpm release:tag desktop --push  # 仅桌面端

pnpm release:tag all --push      # 两端 version.json 相同时，连续 push 两个 tag
```

CI 会校验：**tag 指向 main 上的 commit**，且 tag 版本与对应 `version.json` 一致。

### 4.3 产物与「最新下载」清单

每次发版 Release 含：

- 带版本号文件（如 `BaiShou-v1.2.4-Android.apk`）
- 固定别名（如 `BaiShou-Android.apk`）

CI 还会更新仓库根目录 [`releases/channel.json`](../releases/channel.json)，记录各端**当前最新**下载直链。官网与应用内更新应读此文件，而不是写死版本号：

```
https://raw.githubusercontent.com/foxletters-hq/BaiShou-Next/main/releases/channel.json
```

### 4.4 本地打包（自测）

仍可用本地命令自测，不必等 CI：

```bash
pnpm release:android
pnpm release:desktop:win   # 需 Windows 本机
pnpm release:all
```

### 4.5 CI 手动触发（备用）

GitHub **Actions → Build & Release → Run workflow**：

| 参数             | 说明                                                        |
| ---------------- | ----------------------------------------------------------- |
| `platform`       | `mobile` / `desktop` / `both`（both 要求两端 version 相同） |
| `ref`            | 默认 `main`                                                 |
| `create_release` | 是否创建 Release                                            |

仍需配置 §1.3 的 Android Secrets。

### 4.6 旧流程（本地打 tag v\* 并 gh release create）

仍可用 `gh release create` 手动上传本地产物；推荐改用 §4.2 分端 tag + CI。
